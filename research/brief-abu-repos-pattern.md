## Verified Facts

### A. The GOOD pattern — "define a tool once, adapt to every surface" (portaldot + cdr-kit converge on it)

**1. One framework-agnostic tool shape + one flat registry; transport packages are thin adapters.**
- portaldot `packages/core/src/lib/types.ts:19` defines `ToolDef { name; description; inputShape: RawShape; handler }` where `RawShape = Record<string, z.ZodType>` (the raw `{field: schema}` passed to `z.object()`). `defineTool()` (`lib/tool.ts:10`) wraps a typed handler and `z.object(inputShape).parse()`s the input. The doc comment states the intent verbatim: *"`core` exposes these; the `mcp` package adapts them to MCP tool registrations (raw shape) and the `web` app adapts them to AI SDK tools (`z.object(inputShape)`). Keeps `core` transport-free."*
- The registry is a flat array: `export const allTools: ToolDef[] = [...]` (`packages/core/src/registry.ts:30`). No builder, no factory, no agent argument.
- cdr-kit is the same idea, slightly more mature: `CdrTool { name; description; inputSchema: z.ZodObject; invoke }` (`packages/tools/src/types.ts`), built by `createCdrTools(agent): CdrTool[]` which just `concat`s per-category modules (`tools/src/index.ts`), plus a `toJsonSchema(tool)` helper for the raw-LLM adapters.

**2. The MCP adapter is ~15-30 lines that loop the registry.**
- portaldot `packages/mcp/src/index.ts` is 31 lines: `for (const t of allTools) server.registerTool(t.name, {description, inputSchema: t.inputShape}, async (args)=>{ const res = await t.handler(args); return res.ok ? {content:[{type:"text",text:JSON.stringify(res.data)}]} : {content:[...], isError:true} })`.
- cdr-kit `packages/cli/src/server.ts:30 createMcpServer` does the identical loop using `tool.inputSchema.shape`.

**3. The Vercel AI SDK adapter consumes the SAME registry — this is the "usable elsewhere" mechanism.**
- portaldot `packages/web/app/api/chat/route.ts:40 buildTools()` iterates `allTools`, wraps each in the AI SDK's `tool({ description, inputSchema: z.object(t.inputShape), execute })`, and hands the result to `streamText({ tools })`. **Same source array, two transports** — the literal answer to "how do they make tools usable in both an MCP and a Vercel AI SDK app."
- cdr-kit `packages/vercel-ai/src/index.ts getVercelAITools(agent)` and `packages/openai/src/index.ts getOpenAITools(agent)` do the same — each is ~15-25 LOC over `createCdrTools(agent)`. (cdr-kit also ships langchain/goat/agentkit adapters, each tiny.)

**4. SIGNING — the user-signs / agent-never-executes pattern lives in portaldot, and it is achieved by registering the write tool with NO `execute`.** This is the single most important file for the target:
- `packages/web/app/api/chat/route.ts:20-54`: read tools go in `SERVER_EXECUTED` and get an `execute` (server runs them, no wallet). The write tool (`portaldot_transfer`) goes in `CLIENT_SIGNED` and is registered as `tool({ description, inputSchema })` **with no `execute`** (line 54). Comment line 17-19: *"a single write tool — transfer — which has NO execute, so the browser handles it and the user signs with their injected wallet ('AI proposes, user signs')."*
- Because there's no `execute`, the AI SDK streams the tool call to the client as an `input-available` part with no output. The client (`packages/web/components/app/chat-app.tsx:223-248`) detects `tool-portaldot_transfer`, renders a `<TransferCard>` preview, and on user click calls `signAndSendTransfer(...)` then `addToolOutput({ tool, toolCallId, output })` to feed the signed result back into the conversation (lines 133-145).
- `packages/web/lib/polkadot.ts:23 signAndSendTransfer` is where the **user's wallet signs in the browser**: it imports `@polkadot/extension-dapp`, gets the injected signer via `web3FromAddress`, builds the tx, and `tx.signAndSend(fromAddress, { signer: injector.signer }, ...)`. The agent only ever proposed `{to, amount}`; it never sees a key. The system prompt reinforces this (`route.ts:60`): *"call the tool — the user will review and sign in their wallet."*
- Important nuance to carry into the Sui build: the tool's **core handler still contains server-side signing** (`packages/core/src/tools/transfer.ts:18 getSigner()` → headless `KeyringPair` from `~/.portaldot-mcp/config.json`, `chain/wallet.ts:30`). So on the **MCP surface** the server signs; the **web surface** suppresses that by omitting `execute`. The signing model is *surface-selected*, not baked into the tool. For the target ("user signs everywhere / agent never holds keys"), the clean DeepBook version is: the tool handler **builds and returns an unsigned tx (a "sign-this" payload)** rather than signing — then both MCP and web return that payload and the user's wallet signs. portaldot proves the web half of this; neither portaldot nor cdr-kit/pacifica ships a pure "return unsigned tx" handler, so that piece is net-new.

**5. Generative-UI returns: tools return typed structured JSON; the client maps `tool-<name>` → a dedicated React card.**
- Returns are plain structured objects via a `Result<T>` discriminated union (`packages/core/src/lib/result.ts`: `{ok:true,data}` | `{ok:false,error}`). On the web side the data object becomes the tool-part `output`.
- `chat-app.tsx:259-286` switch-maps each tool name to a component: `tool-portaldot_get_balance → <BalanceCard>`, `…_validators → <ValidatorsCard>`, `…_staking_info → <StakingInfoCard>`, etc., with `<SkeletonCard>` while streaming and `<ErrorCard>` on `{error}`. The flagship `<TransferCard>` (`components/tools.tsx:700`) is a receipt-style confirm card with `Authorize`/`Cancel` buttons, an `AWAITING SIGNATURE → SIGNING → FINALIZED` status, identicons/QR, and post-confirm tx/block explorer links. **This is generative UI driven purely by the tool's typed output shape** — the exact thing the target wants (rich structured returns the frontend renders as components). Typed input/output interfaces (`TransferInput`, `TransferOutput`) are co-located with the card.

**6. Config / install DX (the cdr-kit + pacifica pattern, relevant to the zero-config requirement):**
- Thin single-bin `npx` package wrapping a `createMcpServer` factory: pacifica `packages/mcp/src/index.ts` documents *why* (lines 9-15): npx needs a dedicated single-bin package so `claude mcp add pacifica npx @pacifica-dev/mcp` "just works." cdr-kit `packages/mcp/src/index.ts` mirrors this and adds `--help/--version` short-circuits *before* any wallet I/O.
- Wallet/network config via env-or-auto-create: cdr-kit reads `CDR_PRIVATE_KEY` else auto-creates `~/.config/cdr-kit/wallet.json`, with `CDR_NETWORK` / `CDR_RPC_URL` selectors (`packages/mcp/src/index.ts` help text + `loadOrCreateWallet`). pacifica: `PACIFICA_PRIVATE_KEY` env else auto-generates `~/.pacifica-mcp/config.json` at `0600` and prints a first-run banner (`cli/src/lib/wallet.ts:21`). portaldot: `PORTALDOT_SEED_PHRASE` env else auto-creates `~/.portaldot-mcp/config.json` (`chain/wallet.ts:30`). All three are zero-config-on-first-run — but note these are the **agent-holds-keys** path; the DeepBook target should keep the env-or-auto config ergonomics but for read-only RPC/indexer config, not a signing key.

**7. CLI + skill surfaces are present.**
- pacifica ships a full `commander` CLI (`packages/cli/src/cli.ts`, 1254 lines) with one tool-per-file under `cli/src/tools/`, plus a `SKILL.md` (`packages/skills/pacifica/SKILL.md`) with a "When to use / When NOT / Tool selection table" — installable as a Claude skill. portaldot ships `packages/skills/portaldot/SKILL.md` + a `/docs/tools` catalog (`web/lib/tools-catalog.ts`) whose `ToolEntry` carries a paste-ready `prompt` and a `wallet?: boolean` flag. cdr-kit ships a CLI (`packages/cli`) and 7+ skills under `skills/`.

### B. The mPilot ANTI-PATTERN (what NOT to do)

mPilot pursued the *same correct principle* ("one framework-agnostic tool registry; thin adapters" — its `ADR-014`) but **exploded it into a full SDK/framework**. Concretely:

- **~30 workspace packages** for what portaldot does in 4. `pnpm-workspace.yaml` globs `packages/*`, `packages/providers/*`, `apps/*`, `examples/*`. The repo's own audit admits the scope: *"PRD + ADR-014/015/018 lock 15 packages: foundation (shared, agent, tools) + 7 providers + 5 adapters (vercel-ai, openai, langchain, agentkit, mcp) + 2 React + ui + skill + sdk"* (`docs/AUDIT-2026-06-14.md:26`) — and the actual tree has even more.
- **A bespoke tool framework on top of the AI SDK that already does this.** `packages/tools/src/types.ts` defines `ConciergeTool` with dual input+output Zod generics, `ConciergeToolAnnotations` (readOnly/idempotent/openWorld/destructive hints), `UICardId`, `supportsNetwork(chainId)`, `ConciergeAgentLike`, and `ProviderToolFactory = (agent) => ConciergeTool[]`. Compare portaldot's entire equivalent: a 24-line `tool.ts` + a 24-line `types.ts`.
- **A 142-line registry-builder with "ten duties."** `packages/tools/src/createConciergeTools.ts` does runtime Zod duck-typing (`_def.type`), rejects `.transform()/.pipe()` on input *and* output, enforces `ZodObject` on both, chain-gates via `supportsNetwork` with factory-index error attribution, dedups names, etc. portaldot's equivalent is literally `export const allTools = [getBalance, transfer, ...]`.
- **Per-protocol "provider" packages** (`@mpilot/aave-v3-mantle`, `ethena-susde`, `ondo-usdy`, `meth-staking`, `lifi-bridge`, `mantle-dex`, `erc8004`) — 7 separate publishable packages, each just a `ProviderToolFactory`. plus **5 separate adapter packages** (`vercel-ai`, `openai`, `langchain`, `agentkit`, `mcp`) and meta-wrappers `@mpilot/sdk` and `@mpilot/agent` that re-bundle the adapters. The adapter type-gymnastics leak into the code: `packages/vercel-ai/src/index.ts` needs a dual overload + a multi-paragraph comment explaining why generics collapse to `unknown`.
- **Process bloat around a hackathon repo:** `biome.json`, `commitlint`, `osv-scanner.toml`, `.gitleaks.toml`, `.trivyignore`, husky hooks, mainnet deploy runbook/rollback/precheck docs, ADRs 011-018, multi-day AUDIT/RECONCILIATION docs — a 726-line `architecture.md`. The signal that this is the wrong shape: mpilot's own docs cite *"CDR-Kit pattern"* and *"SDK-DX-STUDY §H"* as the thing they're implementing — i.e. they took cdr-kit's clean ~5-file idea and re-derived it as a 30-package platform with formal ADRs.

**The lesson:** the *principle* (single registry, thin per-surface adapters, typed structured returns) is correct and is exactly what portaldot/cdr-kit do in a handful of files. mPilot's failure was packaging that principle as an extensible multi-framework SDK (custom tool primitive + factory registry + per-protocol provider packages + 5 adapter packages + meta-SDK wrappers) instead of a flat array and two ~20-line loops.

## Inferences

- **Replicate portaldot's shape, not cdr-kit's package count.** For a 1-month hackathon build, the winning structure is portaldot's 4 packages: `core` (all tools + registry, transport-free), `mcp` (loop adapter), `web` (Next.js chat + AI SDK route + generative-UI cards), `skills` (SKILL.md). cdr-kit is the proof that the pattern scales, but its 17 packages (separate goat/langchain/agentkit/openai/story/forms adapters) are more than the DeepBook target needs — pull in extra adapters only if a specific judge-facing surface demands it.
- **For "user signs / agent never holds keys" on Sui, evolve portaldot's idea one step:** make the DeepBook write-tool handler **return an unsigned/serialized Sui `Transaction` (PTB) payload** rather than signing. Then the MCP surface returns that payload for a wallet/CLI to sign, and the web surface (mirroring `chat-app.tsx` + `polkadot.ts`) intercepts the no-`execute` tool part and signs with the user's Sui wallet (`@mysten/dapp-kit` `signAndExecuteTransaction`), feeding the digest back via `addToolOutput`. This keeps a single tool definition feeding both surfaces while *never* placing a key in the agent — strictly cleaner than portaldot's surface-selected headless signer.
- **Generative UI maps cleanly to DeepBook Predict:** the `tool-<name> → <Card>` switch in `chat-app.tsx` is directly reusable for a `MintPositionCard` (binary/vertical position confirm-and-sign), `MarketCard`, `VolSurfaceCard`, `PortfolioCard`, `HistoryCard` — fed by the public Predict server data the docs describe as "render-ready market, vault, portfolio, and history data."
- **Read-vs-sign split = the trust story for judges:** keep a `SERVER_EXECUTED` set (oracle/market/portfolio reads from `predict-server.testnet.mystenlabs.com`, no wallet) and a `CLIENT_SIGNED` set (mint/close position), exactly as `route.ts:20-38` does. A per-tool `wallet?: boolean` flag in the catalog (portaldot `tools-catalog.ts`, pacifica `tools.ts`) documents which is which.

## Unknowns And Questions

- **Drift risk in the docs catalog:** both pacifica (`web/lib/tools.ts`) and portaldot (`web/lib/tools-catalog.ts`) **hand-maintain** a second tool list for the marketing/docs site, separate from the executable `allTools`. This duplicates name/description and can drift. Unknown whether either repo generates the catalog from the registry; for the target, consider deriving the catalog from `allTools` to avoid drift. (Not verified that any of these repos auto-generate it — they appear hand-written.)
- **No repo ships a true "return unsigned tx" handler.** portaldot's web user-signs path is verified, but the *tool's own handler* still signs server-side (`transfer.ts`/`chain/wallet.ts`). The pure agent-never-holds-keys-on-every-surface variant is an extrapolation, not something copied 1:1 from these repos. It needs to be built and tested against DeepBook on Sui testnet.
- **Sui-specific signing ergonomics unverified here:** the exact `@mysten/dapp-kit` / `@mysten/sui` PTB build-and-sign flow, the DeepBook Predict Move entry functions, and the `predict-server` API shapes were not inspected in this task (out of repo scope) — only the *track facts* were confirmed via [blog.sui.io](https://blog.sui.io/introducing-deepbook-predict/) and [docs.sui.io](https://docs.sui.io/onchain-finance/deepbook-predict/). Those should be researched directly before architecting the tool handlers.

**Bottom line:** Replicate **portaldot-mcp's** `core(registry) → mcp(loop) + web(AI-SDK route + generative-UI cards)` shape, borrow **cdr-kit's** thin-adapter discipline and env-or-auto config DX, adopt portaldot's **no-`execute` write tool → browser-wallet-signs → `addToolOutput`** loop as the user-signs mechanism (upgraded to return an unsigned Sui PTB so the agent holds no key on any surface), and explicitly **avoid mPilot's** custom `ConciergeTool` framework, 142-line "ten-duty" registry builder, per-protocol provider packages, and 5 separate adapter packages.
