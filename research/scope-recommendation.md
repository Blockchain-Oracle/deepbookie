# Scope Recommendation & Feasibility — DeepBook Agent (V3 Spot + Margin + Predict)

## Recommendation

Go MEDIUM-WIDE, Predict-weighted. v1 = ~27 tools across 3 primitives, but the registry is unmistakably a PREDICT agent with Spot/Margin as supporting context. Concretely: ship ALL of Predict's user surface (the moat — every tool here is HIGH novelty and only available because you hand-build the PTBs), a TIGHT Spot slice that exists only to (a) prove the one-registry/SDK-wrapper architecture and (b) serve a real user need — get/route dUSDC to fund a Predict bet and read live book/price context — and a MINIMAL Margin slice (read-only + one borrow path) as a clearly-labeled "stretch" that demonstrates leverage-into-Predict without becoming a second product. Do NOT dump the full ~50-tool Spot SDK surface (governance, flash loans, trade-caps, all 6 swap variants, every level-2/order-introspection read); that breadth is hollow — it is low-novelty SDK passthrough that buries the 8 novel Predict tools and signals "we wrapped a library" instead of "we built the first Predict client." Breadth comes from Predict DEPTH (binary + range + LP, full lifecycle) plus a thin, purposeful Spot bridge — not from Spot tool-count. Total reasonable v1: 18 core + 9 nice-to-have = 27; everything else is explicitly out.


**Tool count:** core-v1 ≈ **18**, with nice-to-haves ≈ **27**.

## Architecture (one registry wraps both)

ONE flat, transport-free registry is the whole trick: `export const allTools: ToolDef[]` where `ToolDef = { name, description, inputShape: ZodRawShape, surface: 'predict'|'spot'|'margin', kind: 'read'|'write', handler }` (the portaldot shape, NOT an mPilot SDK-around-it). The handler contract is uniform regardless of how the tx is built: READ handlers have an `execute` that calls the indexer/devInspect and returns typed structured data; WRITE handlers have NO `execute` — they build and return an UNSIGNED Sui `Transaction` (PTB) as the tool output, and signing is pushed to the edge (local managed keypair for MCP/CLI, browser wallet via @mysten/dapp-kit-react for web). This uniform 'reads execute, writes return unsigned PTB' contract is what lets SDK-backed and hand-built tools coexist with zero special-casing in the adapters. The SOURCE of the PTB differs but the registry doesn't care: a Spot write handler calls the official `@mysten/deepbook-v3` curried builders (`client.deepBook.placeX(params) -> (tx) => ...`) and composes them onto the tx; a Predict write handler does raw `tx.moveCall({ target: '<pkg>::predict::mint', typeArguments: [DUSDC_TYPE], arguments: [...] })` because no SDK exists. Both return the same `Transaction` object. Reusability/isolation: keep Predict as its own self-contained module — `tools/predict/` exports its own `predictTools: ToolDef[]` plus a standalone `predictClient` (the hand-built PTB builders + indexer-REST/devInspect readers + the SVI->N(d2) curve math), depending only on `@mysten/sui`, never on `@mysten/deepbook-v3`. Spot/Margin live in `tools/spot/` and `tools/margin/` wrapping the SDK. The top-level registry is just `allTools = [...predictTools, ...spotTools, ...marginTools]`. Because `predictClient` has no dependency on the registry or the SDK, it is independently publishable as 'the first DeepBook Predict TS client' — the durable artifact. Two ~25-line adapters consume `allTools`: an MCP adapter (loop -> `server.registerTool`) and a Vercel-AI adapter (loop -> `tool({ inputSchema: z.object(t.inputShape), execute })`); the web app imports the same array in-process (skip the network MCP hop for v1). A `surface`/`kind` tag on each ToolDef lets any adapter filter (e.g. expose only Predict + Spot-fund in the demo, hide Margin stretch) without touching tool code.


## Rationale

The research is unambiguous on three facts that drive the scope: (1) Spot+Margin already have a full official SDK -> low effort but LOW novelty (a judge sees a library re-export); (2) Predict has NO client at all -> every tool is hand-built and HIGH novelty, and it's first-of-kind on two axes (first Predict MCP, first Predict TS client); (3) the prize is 50% real-world + 20% UX, both of which live in the genUI web app's Predict flow. 'More tools = more capable' is true only if the tools are differentiated — 50 Spot SDK passthrough tools add capability the judge attributes to Mysten, not to you, and they bury the 8 tools that are actually yours. So the winning shape is DEPTH in Predict (all writes + the reads that feed the four genUI widgets: vol-surface curve, range payoff, vault/PLP, live PnL) and a SURGICAL Spot bridge whose entire justification is 'you need dUSDC to place a bet' + 'show live BTC/price context for reasoning' + 'prove the one-registry architecture wraps the SDK too.' That's ~5 Spot tools, not ~50. Core-v1 (18) delivers the complete headline demo end-to-end on testnet: create_manager -> swap-to-dUSDC -> read odds/curve/vault -> mint binary or mint_range -> redeem, with live PnL. The nice tier (+9 -> 27) closes the range/LP lifecycles, adds the novel keeper redeem_permissionless, and read-only margin context — all low effort because the PTB/SDK scaffolding already exists. Everything in the stretch tier (the ~40-tool Spot bulk: governance, flash loans, full order book, trade-caps, all swap variants, and the Margin borrow machinery) is explicitly OUT of v1 and is a one-line registry add later if a real story ever needs it. This maximizes a REAL, novel, demoable agent: breadth comes from covering Predict completely (which only you can do) plus a thin purposeful bridge — never from inflating Spot tool-count, which would dilute the exact thing that makes this a winner.

# Feasibility — Adversarial Check

**Verdict:** `tight-but-yes`


**Dilution check (does all-V3 weaken the Predict moat?):** Going all-of-V3 DILUTES the moat — do not do it. The defensible novelty is narrow and specific: first Predict MCP / first Predict TS client at all (confirmed: SDK has zero predict refs) + the agent reasoning over the SVI vol surface to propose the RIGHT primitive (binary vs range) with implied probability and breakeven, user-signs-no-key. That moat lives entirely in Predict. Spot and Margin already have an official mature SDK (@mysten/deepbook-v3 covers them fully) and a read-only precedent MCP (ExpertVagabond/sui-mcp-server) — so a 'whole DeepBook stack agent' is LESS novel, not more: you'd re-wrap a solved surface and burn the hours that should deepen the one surface nobody has touched. The 'whole stack' story sounds bigger but reads as unfocused to a Predict-track judge; the sharp story ('the only way an agent can trade Predict, and it reasons about the vol surface') reads as inevitable. The ONLY non-diluting role for Spot/Margin is a thin read-only context layer (e.g. show the DeepBook spot BTC price the Predict oracle settles against) — strengthens the 'grounded in real DeepBook data' narrative for ~an hour of work. Anything beyond that is the mPilot mistake the team already diagnosed: surface area mistaken for depth.

## Cut lines (what goes first if time-pressed)

1. Voice + genUI coordination (already marked STRETCH — cut first, zero hesitation)
2. All Spot + Margin WRITE tools — expose none; the track is Predict, mainnet writes don't belong in a testnet demo
3. Conversation / genUI history persistence (Postgres parts[] store + transaction table) — nice durability story but not demo-critical; degrade to session-only in-memory if time bites
4. Spot/Margin READ tools beyond a single 'spot price context' read — the 60-tool catalog is a trap
5. Range trading (mint_range / redeem_range / range-payoff widget) — keep binary mint/redeem as the hero; range is the impressive-but-droppable second act
6. LP path (supply/withdraw + vault card + PnL timeseries) — real and demoable but secondary to mint; cut withdraw + PnL polish before cutting mint
7. Enoki sponsored-tx + keystore encryption — strong sponsor signals but pure upgrades; cut without touching the core demo
8. MCP network transport (Streamable HTTP) + OAuth — ship in-process registry import for web, stdio for local; the hosted-MCP hop is post-hackathon

## Sharpest version to commit to

Commit to Predict-deep, not V3-wide. One registry of ~5 Predict write tools (create_manager, mint, mint_range, redeem, supply) + Predict read tools (indexer REST + on-chain getters + build_curve), plus AT MOST one Spot read for settlement-price context. Headline = the generative-UI chat-to-sign web app where the live probability smile (SVI -> N(d2)) is the primary surface and the mint sign-card emerges from clicking a strike (shows implied probability + breakeven; user signs in their own browser wallet via the NEW @mysten/dapp-kit-react; agent holds zero keys). MCP + CLI + skill are the same in-process registry behind thin adapters — the 'any agent can now trade Predict' durability layer, not co-equal demos. Day-0 gating tasks BEFORE any feature code, in exact order: (1) confirm dUSDC actually arrives from the tally faucet into a testnet wallet; (2) hand-build + sign + execute one real predict::mint on testnet and capture the digest, copying deposit.ts arg ordering. If both land, this is ship-real-yes. Until they do it's tight-but-yes — because the entire 'real end-to-end testnet trade' claim rests on those two unproven seams; everything else (portaldot registry pattern, genUI mechanism, dapp-kit signing) is already de-risked by working reference code.
